# スクリーニング基準: デコンパイル研究 v3（統合コーパス用）

## 目的とスコープ

本スクリーニングは、**デコンパイル・パイプライン**（バイナリから高水準コードへの復元処理）に対して貢献する研究を、**手法を問わず**統合的に収集するためのものである。

v2までとの主な違い:
- **ML/DL必須条件を撤廃**: 古典的手法・形式的手法も対象とする
- **スコープ定義をパイプライン基準に変更**: 「パイプラインのどの段に貢献するか」で判定する
- **評価・基盤・知見の研究も採択対象**: 手法提案に限定しない

設計の根拠:
- タスク中心（手法不問）のスコープ設計は Haq & Caballero (ACM CSUR 2021)、DecompileBench (2025) 等に倣う
- 除外条件の構造は Hou et al. (TOSEM 2024) の EC 設計および Kitchenham SLR ガイドラインに倣う

---

## デコンパイル・パイプライン（IC1 の判定基盤）

以下の9段をデコンパイル・パイプラインと定義する。

```
Binary (machine code)
  → [1] Decoding / Disassembly        … 命令デコード、コード/データ判別、関数境界検出
  → [2] Lifting to IR                 … 機械語→中間表現への変換
  → [3] CFG Recovery & Structuring    … CFG回復、if/while/switch等の制御構造化
  → [4] Dataflow / SSA / Simplification … 値追跡、式回復、式簡約
  → [5] Memory / Variables / Calling Convention … スタック変数、引数、呼出規約の回復
  → [6] Type Recovery                 … 型・構造体の推定
  → [7] Symbols / Names               … 関数名・変数名・識別子の回復
  → [8] Codegen / Presentation        … 擬似コード/AST/ソースコード出力、可読性
  → [9] Evaluation                    … 正しさ・可読性・作業効率の評価（横断的）
```

パイプラインの途中段（IR、アセンブリ、デコンパイラ出力等）から開始する研究も対象に含む。

## 採択条件

### IC1: デコンパイル・パイプラインのいずれかの段に対して貢献がある

論文が上記パイプラインの**いずれかの段**に対して、以下のいずれかの形で貢献していること:

| 貢献の種類 | 説明 | 例 |
|---|---|---|
| 手法 | その段の処理を行う手法の提案・改善 | Cifuentes (構造化)、LLM4Decompile (E2E復元)、TIE (型推論) |
| 評価 | その段の評価枠組み・ベンチマーク・比較 | DecompileBench、Fidelity Issues Taxonomy |
| 基盤 | その段を支えるIR設計・ツール基盤・検証手法 | SLEIGH/p-code、microcode、lifter testing |
| 知見 | その段に関するユーザスタディ・実態調査・体系的整理 | RE作業の実態調査、可読性調査 |

> **判定の核心**: 「この論文はパイプラインのどの段の話か？」に答えられれば採択候補。答えられなければ除外候補。

## 除外条件（いずれか該当で除外）

### EC1: パイプラインの消費者

デコンパイラの出力を**利用する**が、パイプライン自体の**改善に貢献しない**研究。

典型例:
- デコンパイラ出力を特徴量にした脆弱性検出
- デコンパイラ出力を使ったコード検索・類似度マッチング
- デコンパイラ出力を使ったマルウェア分類・ファミリ識別
- デコンパイラ出力を使ったパッチ検出・差分識別
- デコンパイラ出力を使った等価性検証

> 判定の目安: 評価指標が F1/AUC/検出率/分類精度中心 → EC1 の可能性が高い。
> 方向性で判断: 「デコンパイル for X」は除外、「X for デコンパイル」は採択。

### EC2: スコープ外の対象

ソフトウェアバイナリのデコンパイル・パイプラインとは**本質的に異なる対象**を扱う研究。

典型例:
- 量子回路の「デコンパイル」
- ハードウェア記述言語の「デコンパイル」
- 3Dモデル・信号処理等における「decompile」の用法

> 注意: パイプラインの途中段（IR、アセンブリ、デコンパイラ出力）を入力とする研究は EC2 に該当しない。

### EC3: 実質的言及のみ

デコンパイルに**言及している**が、手法・評価・基盤・知見のいずれの記述も**不十分**な研究。

典型例:
- 「将来デコンパイルに応用したい」程度の言及
- 関連研究としてデコンパイルに触れているだけ
- デコンパイルを背景説明に使うだけで、貢献がパイプライン外にある

> Hou et al. (TOSEM 2024) EC8「LLM の使用に言及しているが、技術の記述がない」に対応。

### EC4: 品質不足

技術的内容が十分でない論文。

典型例:
- ポスター、1-2頁のデモ
- エディトリアル、ニュースレター
- 技術的記述が極めて薄い短報

> 出版形態ではなく技術的実質で判断する。博士論文・テクニカルレポートでも技術的に十分な記述があれば採択する（例: Cifuentes 博論、Van Emmerik 博論）。

### EC5: 重複

同一研究の異なるバージョンが複数存在する場合、**最も完全なバージョン1つのみ**を残し、残りを除外する。

対象:
- arXiv プレプリント + 出版版 → 出版版を残す
- 短縮版 + 拡張版 → 拡張版を残す
- 同一著者による段階的改良版 → 最新版を残す

### EC6: 全文アクセス不可・非英語

- 全文が取得できない
- 英語以外で書かれている

## 判定の注意点（運用ルール）

### 1) 「decompile」の語の有無で判断しない
- タイトルに「decompile」があっても、EC1〜EC6 に該当すれば除外
- タイトルに「decompile」がなくても、パイプラインへの貢献があれば採択（例: 型推論、制御構造化、関数境界検出）

### 2) コンパイラ理論の古典はコーパス外
- SSA、interval analysis、構造化定理などのコンパイラ側の基礎理論は、パイプラインへの「直接の貢献」ではないため IC1 を満たさない
- 歴史パート本文では「理論的基盤」として引用する（コーパス外の参考文献として扱う）

### 3) 境界例の判定
- **採択寄り**: 変数名復元、型推定、制御構造回復、IR設計、デコンパイラ比較、可読性調査 → いずれもパイプラインの特定の段に対応
- **除外寄り**: バイナリの埋め込み学習が主で最終成果が検索/検出（EC1）、解析パイプラインの一部としてデコンパイラを利用するだけ（EC1）

## 分類タグ（採択後に付与）

採否判定とは独立に、採択された論文に以下の3軸のタグを付与する。

### 軸1: パイプライン段（複数可）

| タグ | 対応する段 |
|---|---|
| `P1-decode` | [1] Decoding / Disassembly |
| `P2-lift` | [2] Lifting to IR |
| `P3-cfg` | [3] CFG Recovery & Structuring |
| `P4-dataflow` | [4] Dataflow / SSA / Simplification |
| `P5-memory` | [5] Memory / Variables / Calling Convention |
| `P6-type` | [6] Type Recovery |
| `P7-name` | [7] Symbols / Names |
| `P8-codegen` | [8] Codegen / Presentation |
| `P9-eval` | [9] Evaluation |
| `P0-e2e` | 全段（エンドツーエンド） |

### 軸2: 手法（1つ）

| タグ | 内容 |
|---|---|
| `M-rule` | ルールベース / パターンマッチ / ヒューリスティック |
| `M-formal` | 形式的手法 / データフロー解析 / 抽象解釈 / 型システム |
| `M-stat` | 統計的手法（DL以前の学習） |
| `M-neural` | ニューラルネット / DL（非LLM） |
| `M-llm` | LLM / 大規模言語モデル |
| `M-hybrid` | 上記の組み合わせ |
| `M-na` | 手法提案ではない（評価・整理・調査が主） |

### 軸3: 貢献種別（1つ）

| タグ | 内容 |
|---|---|
| `C-method` | 手法の提案・改善 |
| `C-eval` | 評価枠組み・ベンチマーク・比較 |
| `C-infra` | 基盤・ツール・IR設計 |
| `C-empirical` | 実態調査・ユーザスタディ |
| `C-survey` | サーベイ・体系的整理 |

## 判定ラベル（記録用テンプレ）

```
- Decision: include / exclude / unsure
- Pipeline stage: P1 / P2 / ... / P9 / P0-e2e（IC1 の根拠）
- Contribution: C-method / C-eval / C-infra / C-empirical / C-survey
- Method: M-rule / M-formal / M-stat / M-neural / M-llm / M-hybrid / M-na
- Reason (exclude): EC1 / EC2 / EC3 / EC4 / EC5 / EC6
- Note: 境界例なら判定根拠を1行
```

## v2 からの変更点まとめ

| 項目 | v2 | v3 |
|---|---|---|
| スコープ定義 | ML/DL を用いた復元研究 | パイプラインへの貢献（手法不問） |
| ML/DL 必須 | **必須**（IN1） | **不要**（手法は分類タグ） |
| 出力の条件 | 高級言語 / 擬似コード高級化 | パイプラインのいずれかの段への貢献 |
| IR 止まりの研究 | 除外（EX-B1） | **採択**（[2] Lifting to IR） |
| disassembly 研究 | 除外（EX-B2） | **採択**（[1] Decoding） |
| 評価・ベンチマーク | 原則除外（EX-C） | **採択**（[9] Evaluation） |
| サーベイ・整理 | 除外（EX-C1） | **採択**（C-survey） |
| 基盤・IR 設計 | 対象外 | **採択**（C-infra） |
| 除外条件の構造 | EX-A/B/C | EC1〜EC6（番号統一） |
| 消費者の除外 | EX-A（目的が別タスク） | EC1（パイプラインの消費者） |
| スコープ外対象 | EX-B4 の一部 | EC2（パイプラインと無関係な対象） |
| 実質的言及のみ | 該当なし | EC3（**新設**） |
| 品質不足 | EX-C の一部 | EC4（技術的実質で判断に変更） |
| 重複 | 該当なし | EC5（**新設**） |
| アクセス・言語 | 該当なし | EC6（**新設**） |
