# スクリーニング基準：デコンパイル研究 v4

## 0. 判定の順番（Decision Flow：迷ったらここに戻る）
1) **ML/DLが中核か？**（IN1）→ いいえ：❌Exclude（EX-IN1）  
2) **目的が「デコンパイル／可読性向上（高級化）」か？**（IN2）→ いいえ：❌Exclude（EX-A*）  
3) **主入力・主出力が範囲内か？**（IN3）→ いいえ：❌Exclude（EX-B*）  
4) **論文タイプが対象外か？**（EX-C*）→ はい：❌Exclude  
5) 判定できない：❓Unsure → 全文確認（Stage3）

> **用語**  
> - **デコンパイル**：binary/asm/既存デコンパイラ出力などの低水準表現から、**高級言語ソース相当**（またはそれに近い構造化表現）へ復元・高級化すること  
> - **可読性向上**：単なる整形ではなく、**ソースに近づく復元・修復**（例：制御構造回復、型推定、変数名復元、式の簡約など）を指す

---

## 1. 目的とスコープ
本スクリーニングは、**機械学習（ML/DL）を中核に用いて**、次を主目的とする研究を対象とする：

- **(a) デコンパイル**：バイナリ／アセンブリ等から高級言語のソースコード相当を復元する  
- **(b) 可読性向上（高級化）**：デコンパイル擬似コード等を、**ソースに近づける**方向に修復・高級化する

ここでいう「ソースコード相当」には、次を含む：
- 高級言語ソース（C/C++/Rust 等）
- **構造化された擬似コード**（制御構造・式・型などが復元/改善されソースに近い）
- **AST / 高レベルIR**（ソース再構成・高級化が目的として明確な場合）

一方、**解析目的の低レベルIRへの単純なlifting（IR-only）**は対象外（EX-B1）。

---

## 2. 採択条件（すべて満たす）
### IN1: ML/DL を中核に用いている
以下のいずれかを満たす：
- Transformer / LLM / Language Model / Neural Network / Deep Learning / Machine Learning を**主要要素**として利用
- **中核判定チェック（簡易）**：  
  - その研究の新規性・性能の主因がML/DL部分にある、または  
  - ML/DLを外すと提案が成立しない（単なる補助・周辺ではない）

> ルールベース／統計のみで、ML/DLが主要要素でない場合は ❌Exclude（EX-IN1）

---

### IN2: 主目的が「デコンパイル」または「可読性向上（高級化）」である
研究の主目的が以下のいずれか：
- **高級言語ソースの復元**（例：C/C++/Rust 等のコード生成）
- **擬似コードの修復・高級化（可読性向上）**  
  - 例：制御構造回復、goto除去、型推定、変数名復元、式の簡約、構造化、API呼び出しの高級化 等

> **注意（重要）**：  
> - **formatting-only（インデント整形・コメント付与・色付け等のみ）**で、構造/意味の復元がない場合は ❌Exclude（EX-B2）

---

### IN3: 主入力・主出力が範囲内である（複数可）
> 「入力1つ・出力1つ」の制約は設けない。論文は複数入力・複数出力を取り得る。  
> ただし、**主入力（primary input）**と**主出力（primary output）**が下記の範囲を満たすこと。

**主入力（いずれか）**
- binary（機械語バイナリ）
- assembly（逆アセンブル結果を含む）
- decompiler output（既存デコンパイラの擬似コード）
- （補助入力として）CFG/DFG/型情報/デバッグ情報/ライブラリ知識/実行トレース等が付随するのは可

**主出力（いずれか）**
- 高級言語ソースコード（C/C++/Rust等）
- 擬似コードの修復・高級化（「ソースに近づける」変換）
- AST / 高レベルIR（ソース再構成・高級化が目的として明確な場合）
- （補助出力として）型注釈・変数名・制約・検証結果等が付随するのは可

---

## 3. 除外条件（いずれか該当で除外）
### EX-IN1: ML/DLが中核でない
- ルールベース中心、統計のみ、ML/DLが周辺的（特徴量や補助に留まる）など

---

### EX-A: 目的が別タスク（デコンパイルは手段）
以下が主目的で、デコンパイル/復元が副次的（特徴量・前処理・説明用）なら除外：
- matching / retrieval（検索・類似度・clone/similarity）
- identification（識別・分類・属性推定）
- SCA（Software Composition Analysis）
- summarization / explanation（要約・説明文生成が主）
- vulnerability detection（脆弱性検出・予測）
- patch detection（パッチ検出・差分識別）
- verification（等価性検証・正当性証明が主目的）

**下位コード**
- EX-A1: classification / identification が主目的
- EX-A2: retrieval / similarity / clone が主目的
- EX-A3: vulnerability / patch / verification が主目的
- EX-A4: summarization / explanation が主目的

> 判定ヒント：評価指標が F1/AUC/検出率/分類精度中心 → EX-A寄りになりやすい

---

### EX-B: 出力が「ソース相当の復元／高級化」ではない
- **EX-B1: IR-only**（解析用の低レベルIR止まりで、ソース再構成/高級化が主目的でない）
- **EX-B2: formatting-only**（インデント/改行/コメント付与等の整形のみで、構造/意味の復元がない）
- EX-B3: 出力が特徴量/埋め込み/グラフ/ラベル等で、コード生成・擬似コード高級化がない
- EX-B4: 対象が本スコープ外（例：量子回路など、ソフトウェアバイナリ復元と別物）

---

### EX-C: 論文タイプが整理中心で、新規復元手法が主でない
- ベンチマーク、データセット、チュートリアル、サーベイ、ツール紹介が中心で、
  **新規のMLベース復元手法の提案・検証が主ではない**場合は除外

**下位コード**
- EX-C1: survey/tutorial/benchmark/dataset/tool overview が中心
- EX-C2: 復元手法の提案はあるが **評価がほぼ無い/極めて弱い**

> 例外（Include候補にできる条件を固定）：  
> - (i) 新規タスク設定が明確  
> - (ii) 強いベースライン提供  
> - (iii) 復元（decompilation/高級化）としての評価がある  
> → 3点すべて満たす場合は Include を検討可（Noteに根拠を書く）

---

## 4. 判定の注意点（運用ルール）
1) “decompile” という語の有無で判断しない（IN/EX優先）  
2) “復元／高級化が主目的か”を最優先（目的が別ならEX-A）  
3) 迷ったら **Unsure → 全文確認**（不足要素をNoteに1行で書く）

---

## 5. 記録用ラベル（運用形）
- Decision: I(include) / E(exclude) / U(unsure)
- Reason code:
  - include: IN1+IN2+IN3
  - exclude: EX-IN1 / EX-A1/2/3/4 / EX-B1/2/3/4 / EX-C1/2
- Note: 境界/不明点（不足している要素、全文確認要否）を1行

---

## 6. 出力形式指定プロンプト（Title/Abstractのみ）
各論文について、**Title と Abstract のみ**を根拠にスクリーニング結果を出力せよ。Markdown記法を用いてよい。**1件あたり5〜12行程度**に収め、長文の解説は避ける。判断は本ルール（IN1–IN3 / EX-*）に従う。

### 出力テンプレ（1論文＝1ブロック）
- **[ID] Title:** （タイトル）
- **Decision:** ✅Include（IN1+IN2+IN3） / ❌Exclude（EX-??） / ❓Unsure（理由）
- **Reason code:** IN1+IN2+IN3 または EX-コード（または不足点）
- **Key evidence (Abstract quotes):**
  - C1(ML/DL): “…”  
  - C2(目的): “…”  
  - C3(入出力): “…”  
  - （Excludeの場合）決定打: “…”  
- **Rationale (2–4 bullets):**
  - （引用に紐づけて、IN/EX該当理由を短く箇条書き）
- **Note (optional, 1 line):** 境界/不明点、全文確認が必要ならその旨

### 引用ルール（必須）
- 引用は **Abstractからの直引用**のみ。各引用は**最長25語程度**、長い場合は…で省略可。
- ✅Includeのときは原則 **C1/C2/C3をそれぞれ最低1つ**引用。
- ❌Excludeのときは支配的な除外理由（EXコード）を裏付ける **決定打引用を1つ**入れる。
- ❓Unsureのときは、確定できない要素（目的/入出力/ML中核など）を引用で示し、全文確認へ回す。
