# 3/3 LLMパート方針（統合コーパス内での扱い）

※ここでも「章構成」や「細かい運用パラメータ」は書きません。
統合コーパスの中で、**LLMパートをどう拾い、どう絞り、どうまとめるか**の方針だけをステップ順に整理します。最後に、あなたが貼った **既存検索ログ（JSON）を“参照用に載せる”**方針も付けます。

---

## ステップ0：位置づけ（LLMパートは何を達成するか）

* LLM/ML（必要に応じて従来のNLPも含む）を用いたデコンパイル研究を、統合コーパスから抽出し、

  * **どのタスク（何を復元するのか）**
  * **どの入力/出力（binary/asm/IR、C/擬似コード/属性など）**
  * **どの評価（指標、ベンチ、再現性）**
    の観点で体系化する。
* LLMパートは原則として **取りこぼしを避ける（網羅性重視）**。

---

## ステップ1：検索（統合コーパスへの取り込み）

### 1.1 方針（確定）

* LLM文献も、歴史文献と同様に **統合コーパスの母集団として拾う**（別建てにしない）。
* 取りこぼし対策として

  * **引用追跡（前方・後方）**
  * **ミニQGS**
    を活用する（全体方針で確定済み）。

### 1.2 使用DBセット（確定）

* **主要DB（4つ）**：ACM Digital Library / IEEE Xplore / Web of Science / arXiv
* **補助DB（2つ）**：NDSS / USENIX（個別に検索する。未実施だが方針として決定）
* ※全体方針（1/3）で確定済み。Springer は参考ログに含まれるが主要DBには含めない。

### 1.3 検索期間・LLM対象の絞り方（確定）

* **検索期間に制限は設けない（全年代対象）**。全体設計と同一の検索で収集する。
* LLMパートの対象文献は、**年代ではなく手法タグで絞り込む**：統合コーパスから `M-neural` / `M-llm` / `M-hybrid` / `M-stat` のタグを持つ論文を抽出する。
* ※旧設計の「2017–2026」という期間制限は撤廃。年代による除外は行わない。

### 1.4 未決（検索実施時に決める）

* 検索語彙の拡張、停止条件、QGS規模、ログ粒度などの詳細設計
  ※方針としては「既存の試行（検索ログ）を参照しつつ統合コーパス向けに再設計する」。
  ※これらは検索実施時に決定する。

## ステップ2：統合（コーパス内での"LLMタグ付け"）

### 2.1 方針（確定）

* 統合コーパスに入った論文は、パイプライン基準の3軸タグで分類して管理する（全体設計 ステップ3.3参照）:
  * **パイプライン段タグ（P1〜P9, P0-e2e）**: どのコンポーネントを扱うか
  * **手法タグ（M-rule / M-formal / M-stat / M-neural / M-llm / M-hybrid / M-na）**: LLMパートの抽出には `M-neural`, `M-llm`, `M-hybrid`, `M-stat` でフィルタ
  * **貢献種別タグ（C-method / C-eval / C-infra / C-empirical / C-survey）**
* LLMパートで扱う論文 = 手法タグが `M-neural` / `M-llm` / `M-hybrid` / `M-stat` のいずれかを持つもの

### 2.2 交差（歴史×LLM）の扱い（確定方針）

* 同一論文が「LLMパート」かつ「歴史の節目」になり得るため、

  * **同一レコードに複数タグを付ける**（二重カウントしない）
  * 歴史側は「代表選定」で説明、LLM側は「分類表」にも載せる
    という形で両立させる。

---

## ステップ3：スクリーニング（LLM文献の拾い方）

### 3.1 方針（確定）

* LLMは **"全て扱いたい"**方針なので、統合コーパスの中でLLM系は原則取りこぼさない。
* 統合コーパスのスクリーニング基準は **スコープベースのIC/ECを採用**（全体設計 ステップ3.2参照）:
  * **IC1**: デコンパイルまたはその構成要素技術を、ソフトウェアバイナリを対象として扱っている
  * **IC2**: 手法の提案、評価枠組みの提案、ツール・基盤の設計、実証的調査、または体系的な分類・整理のいずれかで技術的貢献を行っている
  * **IC3**: 全文がアクセス可能であり、英語で記述されている
  * **EC1**: デコンパイラ出力を下流タスクの入力として利用するのみで、デコンパイル自体の改善には貢献しない
  * **EC2**: ソフトウェアバイナリのデコンパイルとは本質的に異なる対象を扱っている
  * **EC3**: 技術的内容が不十分である
  * **EC4**: 同一研究のより完全なバージョンが存在する
* LLMパートの対象文献は、上記で採択された論文のうち手法タグが `M-neural` / `M-llm` / `M-hybrid` / `M-stat` のもの

### 3.2 旧未決事項の解決状況

## 参考：既存検索ログ（JSON）を文書に載せる方針（決定）

### 位置づけ（確定）

* あなたが貼ったJSONの検索は、**LLMパート単体を想定して練った試行**であり、統合コーパスの正式ベースラインとしては固定しない。
* ただし、今後の再設計で参照できるように、文書に **「参考：既存検索ログ」**として掲載する。

### 掲載する内容（確定）

* DB名、検索クエリ、検索日、対象期間、件数、URL（あるものはURLも）を残す。
* その上で、文書では「これは参考ログであり、統合コーパス用に再設計する」ことを明記する。

### 参考ログに含まれるDB（あなたのJSONより）

* **ACM / IEEE Xplore / Web of Science / arXiv / Springer**
* クエリ種別：

  * `decompil*`（title/abstract中心）
  * 拡張クエリ（query2：復元系語彙×バイナリ語彙×ML/LLM語彙）

### 予定（確定・未実施）

* **NDSS と USENIX は個別検索する予定**（未実施）
